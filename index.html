HTML? Who's that?

HTML is a convenient acronym (there are A LOT of acronyms in software) for Hyper Text Markup Language. I will be referring to this as HTML from hereon out because I'm sane.

Basically, you use HTML to write up what goes on your webpage from text and images to actual areas containing these elements. You can think of the web coder at this point as an artist deciding where his painting should go and what content should go on the canvas.

Broadly, here's what HTML consists of and what you will be seeing later when you get to work with it!

  • HTML is a series of semantic elements that tell an Internet browser how to display your content
  • Every element is represented by special tags which we will get to learn about more in-depth later
  • Tags label the content that you put onto the webpage

HTML's current version, HTML5, emphasizes a "standard" way of declaring elements in our document such that writing and reading the code is much easier to achieve the same results of what we want to write.

Here is an example of a purely HTML page. It might remind you of an old '90s era website.

CSS? Sounds Fancy.

CSS stands for Cascading Style Sheets. Like the title implies, this is the tool we use to style our webpage: CSS designates the overall layout of the HTML elements and what they look like. The web coder as an artist is using his paint strokes and arranging the designs on his painting--however, CSS goes a lot deeper than this and there are some incredible designs and layouts you can make of it that you will get to create!

Broadly, here's what CSS consists of:

  • A wide variety of attributes and characteristics you can apply to your elements (e.g. color, spacing, font size, padding)
  • We will see this more later: By nature of design, you can inherit of share these characteristics between elements by only having to write a CSS statement once
  • Inline: You style the look and layout of each element individually
  • Internal: You create a special section in your webpage which decides the overall style of your webpage
  • External: You create a special file outside of your HTML document that is connected to it and still styles the HTML elements

Here is an example of CSS at work!

Distinguishing HTML and CSS

If HTML forms the structure of our document, then CSS should provide the presentation of it. HTML does not so much focus on the style as what information we are trying to convey to a user and CSS is the opposite. This is mainly a design philosophy called semantic HTML that we should be aware of as we develop our project.

In particular, HTML5 was created with this distinction in mind with its variety of semantic features for writing HTML and many of the methods we'll be learning about for HTML are a result of this.

The Basic, Minimal Structure of HTML

Right now, you are looking at the bare minimum for an HTML page to run. If you preview your page with the Show button then you should see a blank, white page. If you look up at your tabs, the tab title will be Hello!.

You won't see this metadata show up on your page, but it can be super useful--there are coders whose entire job is to optimize for search engine relevancy!

  • <body> tag is where things get interesting for building visual elements since it captures all the visible content that we will provide

Tags and Their Attributes

We're about to start handling the big work of making our portfolios, but there are a couple more topics that are crucial to understand before we begin.

I've mentioned tags a few times and you might be wondering what those are. Tags are essentially labels for our elements, denoted by angle brackets. At this point, you've already seen several appear in our basic document and I'd like for you to get an idea of how to write them as well as a few exampels that we will be using to make our portfolio.

  • Most tags come in pairs with the closing tag containing a /. An example of this is <title>...</title> or <body>...</body>
  • This starting and ending tag formulation will tell the browser where your element section starts and ends with all the content going in-between

And here are a few important tags we'll be working with:

  • <div>: A more general and commonly used tag, this defines a general division or section of our HTML document
  • <section>: Can be used to divide our document into sections and subsections and has replaced the <div> to divide up semantic information for use cases like pairing headings with paragraph sections
  • <nav>: This tag defines a set of navigational links which we can provide as separate tags. Before the navigation tag, it used to be much more convoluted to create a navigational menu
  • <h1>, <h2>, <h3>, ... , <h6>: Headings are titles that get displayed on our page. With each level of heading (up to 6) the font becomes smaller
  • <ul>, <li> : Unordered list and list item. If you wrap something as an unordered list you will essentially make a bulleted list. We use a list item to denote individual items in the list
  • <p>, <br />: A paragraph element and break tag. In HTML, you won't be able to change how your text looks the usual way--you can't just add extra spaces or lines in your code--and paragraph tag will help you write the text for browsers. Break tags will create a line break in your document.
  • <a>: The anchor tag supports hyperlinks which will be used to link to other webpages or even a specific area on our own webpage
  • <img>: This tag allows you to insert images into your HTML site. You will specify a file name for it to determine which image is displayed
  • <form>: This tag will create a form with multiple fields that a user on your website can put information in. This will make up the contact form portion of our portfolio

I know this list might seem overwhelming or difficult to remember, so here is a visual example of these tags at work in our browser so you can get a good picture of how they will appear.

I want to mention that there are many different tags than the ones I've listed and that some of them will come into play in our own project. However, the tags I've listed have the most distinct functions in our project so I've chosen to highlight them.

Let's add two of these elements into our document now. Inside our <body> element area, we'll add a <p> enclosing our text and a <div> enclosing both of these elements.


<body>
    <div>
        <p>There can only be one!<>
    </div>
</body>

You won't be seeing any visible changed when you preview your page, but notice how we're actually putting our <p> tag inside of the <div>tag. This is what we call nesting and it plays an important part in how our elements interact together, especially as we begin to look at special characteristics of tags called attributes.

Attributes and Nesting

Conceptually, we can think of all of our elements as parents, children, and descendants. Our <div> is a parent to the <p> element because we declared it inside of the <p> tag. We call this type of relationship nesting. In fact, the <div> and <p> are both nested inside the <body>.

However, if we were to nest another element inside the <p> tag then it becomes a descendant of the original <div>. Our <div> would be a grandparent!

This understanding of relationships between elements is especially important as we start getting into attributes. Attributes are characteristics we assign to HTML elements inside the opening tag like what images will be displayed by the <img> tag, what link or section our <a> tag points towards, or even how we group elements together.

We can embed an image in our document now by adding an <img> tag like so: <img src="link" alt="alt text">.

Navigation Bar

With HTML5, HTML designates a specific <nav> tag for a navigation bar. We can wrap this tag over as many <a> tags as we need for each link in the navigation bar. In this case, we will just be using 3 links to direct the user to 1) the about, 2) the work, 3) the contact.

You'll notice that we have a new attribute in our <a> tag. This href attribute indicates where our link will point to. Contact Section

Before we design our form, we will add one more <section> tag to create our contact section. We'll also add a sub-header inside it to get our contact phrase in.


<section>
    <h2>Let's get in touch!</h2>

We're now at the final step of laying down all our HTML with HTML forms.

HTML forms allow you to collect user input so that you can process the information in the form of blog comments, mailing lists, e-commerce products, and much, much more. Forms have two components attributed to them: a frontend side and the backend server.

The frontend refers to the look and appearance of the form created through HTML and CSS while the backend refers to the code that processes the information given to the form by the user. We won't be focusing on the backend portion and instead we will be looking to design the frontend interface--what the user will see.

In HTML, every form is designated by the <form> tag with two important attributes: action and method. We can add a form into our document now with this code:


<form action='' method="get">
</form>

The action attribute points to a URL that processes all the information we write in our form. Developers will set up special URLs in a web server to be able to deal with this information in a way that fits with the application they are building. However, we leave this blank in order to redirect back to the page we are on when we Submit our information since we are not dealing with any backend technologies.

The method attribute deals in two common types of requests in web technology: post and get. This relates to how the form gets processed in the web server. The way you can think about these requests is that we use post when we are changing data on the server and get when we are getting data.

In our case, we want to use the get request so we can see the information we send to the form.

Input Elements

Unlike other HTML elements, forms include special elements called input elements. Input elements are what we use for users to put their information in like textboxes, email, radio buttons, or textareas.

We'll start with the most common input element: text inputs.

Text Inputs

We'll start off with the name field. Inside our <form> tag we can insert a text field input using this code:


<form>
    <input name="full-name" type="text" />
</form>

You'll notice there isn't really any indication of what our text input is looking for. To address this, HTML has special <label> tags which we can associate with our inputs.

We can attach a label now like this:


<form>
    <label for="full-name-id">Name:</label>
    <input id="full-name-id" name="full-name" type="text" />
</form>

I've introduced a new id attribute between the <label> and <input>. This will allow us to relate the two elements to each other in the <form>. However, we will come to see that ids play a much larger role in styling and associating our styles with those elements.

For now, you can think of the id attribute as giving a unique name to our element. That is, no two elements should have the same id if we choose to assign them. Our text input has a unique name of full-name-id.

Email Inputs

We'll need some way to keep in contact with the user. Email works--as long as it's not spam! We can insert an email input using this code:


<!-- Inside of the form; remainder of code above -->
<label for="email-id">Email:</label>
<input id="email-id" name="email" type="email" placeholder="yang@example.com" />
...

This looks eerily similar to what we did before for our text input. Really, you can think of the email input as another text input that validates whether or not what the user wrote inside of it is an email address (e.g. yang@example.com).

If you're working with Chrome or Safari, then you will only see this validation pop up when we Submit our form. It's similar to when you forget to write your name in an online form, attempt to submit it, only to find an ominous red box pop up on your screen with a message like 'Please type in your name here.'

However, for the sake of understanding why clean inputs are important, you can think about the value of having in-built validation like this. If we were to default to a regular text input instead of using type="email", then a user could accidentally send a non-email address like bademailexample.com or random characters to our server.

Email's not the only validation you can apply to your <input> tags. The Mozilla Developer Network has a bunch of other useful validation methods that you can read about in the <input/> reference.

Also, I've added a new placeholder attribute inside of the tag. You might come across faded text pre-populating web form fields. The placeholder attribute is exactly that. It's some default text that you can have in the field to help the user understand what to type in.

Radio Buttons

Let's say we need the user to choose between a set of predefined options in our form. This is a perfect use case for radio buttons. In our project, we're having the user select their class year.

Radio buttons are declared individually by making an <input> with type="radio".

Radio buttons work in an interesting way in that they have to be grouped together in order to function properly. We can perform this grouping by nesting each of our <input> tags inside of a <fieldset> element, labeled with a <legend>. The <legend> will label the overall group of radio buttons.

Another key idea in grouping our radio button elements is that we will assign the same name attribute to all of them.


<!-- Inside of the form; remainder of the code above -->
<fieldset>
    <legend>Class Year</legend>
    <input type="radio" id="year1" value="2020" name="class-year" />
    <label for="year1">2020</label>
    <input type="radio" id="year2" value="2021" name="class-year" />
    <label for="year2">2021</label>
    <input type="radio" id="year3" value="2022" name="class-year" />
    <label for="year3">2022</label>
    <input type="radio" id="year4" value="2023" name="class-year" />
    <label for="year4">2023</label>
</fieldset>
...

Try clicking on each button to see that it works! You should only be able to select one value at a time just as a radio button should work.

Textareas

We don't leave a lot of room for a user to express themselves with one line of text. So why not multiple lines of text? We can use the <textarea> tag to create a multiline text field for the user to berate these pitiful creative souls.

There are two special attributes associated with the <textarea> element which are rows and cols. As the names suggest, these will adjust the respective width and height of our <textarea>. In CSS, we will learn how to replace these in-line attributes with more flexible dimension specs. For now, we will use them to our heart's content.


<!-- Inside of the form; remainder of the code above -->
<label for="message-id">Message:</label>
<textarea id="message-id" name="message" rows="5" cols="30">
</textarea>
...

Submit this, submit that. Why does this man keep telling me about Submit?

When we submit our form with this <button>, the browser will make sure all of our <input> elements are valid and then submit our information to the URL provided in action.

Let's add the button now:

...

We can see this information because we set our <form> to use the get method. In a full web application, our backend would be able to access this information and use it however our application needs to. That's the magic of forms!
